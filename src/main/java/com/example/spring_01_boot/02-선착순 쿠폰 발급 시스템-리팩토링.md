# 선착순 쿠폰 발급 — 리팩토링 정리

`couponController` 및 쿠폰 **발급** 유스케이스를 과제 문서(`assignment/02-선착순 쿠폰 발급 시스템.md`)와 REST 관례에 더 가깝게 맞추기 위한 변경 사항입니다.

---

## 1. API (컨트롤러)

| 이전 | 이후 |
|------|------|
| `GET /coupons/issue?userId=&couponName=` | `POST /coupons/{couponId}/issue` |
| 응답 없음 (`void`) | JSON `CouponIssueResponse` |
| 쿼리로 쿠폰 식별 | 경로 변수 `couponId` (예: `c-1`) |

- 요청 바디 DTO: `CouponIssueRequest` (`userId`, `@NotBlank`)
- 생성 API `POST /coupons`는 기존과 동일

---

## 2. 서비스 계약

| 항목 | 내용 |
|------|------|
| 시그니처 | `issueCoupon(String couponId, String userId)` → `CouponIssueResponse` |
| ID 해석 | `CouponIdParser`: `c-{Long}` ↔ DB PK |
| 트랜잭션 | `@Transactional` 유지 (발급 전체가 한 트랜잭션) |

---

## 3. 동시성·정합성

| 항목 | 내용 |
|------|------|
| 비관적 락 | `CouponRepository.findByIdForUpdate(id)` — `PESSIMISTIC_WRITE` |
| 목적 | 동시에 여러 발급 요청이 와도 `issuedQuantity` 갱신을 직렬화 |
| 한계 | DB/락 지원 여부는 프로덕션 DB 기준으로 재검증 필요 |

---

## 4. 중복 발급 방지

| 층 | 구현 |
|----|------|
| 애플리케이션 | `existsByUserIdAndCouponName(userId, couponName)` 선검사 |
| DB | `coupon_transaction` 테이블 `(user_id, coupon_name)` **유니크 제약** (`uk_coupon_transaction_user_coupon`) |
| 응답 | `409` + `COUPON_ALREADY_ISSUED` (과제 예시 `errorCode`에 맞춤) |

---

## 5. 비즈니스 규칙·예외 매핑

도메인 예외를 도입하고, `coupon.controller` 패키지용 `@RestControllerAdvice`로 HTTP 상태와 JSON을 고정했습니다.

| 예외 | HTTP | `errorCode` |
|------|------|-------------|
| `InvalidCouponIdException` | 400 | `INVALID_COUPON_ID` |
| `CouponNotFoundException` | 404 | `COUPON_NOT_FOUND` |
| `CouponIssueConflictException` (중복) | 409 | `COUPON_ALREADY_ISSUED` |
| `CouponIssueConflictException` (소진) | 409 | `COUPON_SOLD_OUT` |
| `CouponIssueConflictException` (만료) | 409 | `COUPON_EXPIRED` |

응답 바디: `CouponErrorResponse` (`errorCode`, `message`)

---

## 6. 발급 성공 응답

`CouponIssueResponse` 필드 (과제 예시 정렬):

- `userId`, `couponId`, `userCouponId` (`uc-{CouponTransaction.id}`), `status` (`ISSUED`), `issuedAt`, `expiresAt`

## 8. 아직 하지 않은 것 (과제 나머지)

- `GET /users/{userId}/coupons` (발급 목록)
- 쿠폰 사용 API 및 `USED` 상태
- `UserCoupon` 전용 모델로의 분리 (현재는 `CouponTransaction` 중심)
- `DataIntegrityViolationException` → 409 매핑 (유니크 충돌 시 애플리케이션 예외와 중복될 수 있어 선택 사항)

---

## 9. 마이그레이션 참고

로컬 H2 인메모리는 스키마가 매 실행마다 새로 생성되므로 유니크 제약 추가만으로 충분합니다.  
기존 파일 DB를 쓰는 경우 `coupon_transaction` 스키마 변경이 필요할 수 있습니다.
