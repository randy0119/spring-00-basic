### 📍 개요

`spring_01_boot/src/main/java/com/example/spring_01_boot/user/service` 는  
`user` 도메인에서 **비즈니스 로직(Service 계층)** 을 담당하는 클래스들이 모여 있는 패키지입니다.

컨트롤러가 “요청/응답” 을 다루는 웹 레이어라면,  
서비스는 그 사이에서 **실제 규칙(로직)과 트랜잭션을 처리하는 중간 계층**입니다.

---

### 🧠 '서비스(Service)'란?

스프링에서 서비스는 보통 다음과 같은 책임을 가집니다.

- 여러 엔티티/레포지토리를 조합해 **비즈니스 요구사항을 구현**
  - 예: 회원 가입 시 ID 중복 체크 → 비밀번호 암호화 → 저장
- **트랜잭션 경계**를 잡는 곳 (`@Transactional` 을 붙이는 위치)
- 컨트롤러에서 받는 DTO 를 도메인 엔티티로 변환하거나,  
  도메인 엔티티를 응답용 DTO 로 바꿔 주는 역할을 함께 수행하기도 함

전형적인 흐름:

1. 컨트롤러가 HTTP 요청을 받아 DTO 로 변환
2. 서비스가 DTO 를 받아 **비즈니스 규칙을 수행**하고, 레포지토리를 호출
3. 필요한 경우 여러 도메인/레포지토리를 조합
4. 결과를 다시 DTO/값 형태로 돌려주기

서비스 계층이 존재하면:

- 컨트롤러는 “웹 요청/응답” 에만 집중할 수 있고
- 레포지토리는 “DB 접근” 에만 집중하며
- **비즈니스 규칙은 서비스에 모여** 코드가 더 읽기 쉬워집니다.

### 🧩 서비스 인터페이스와 구현체 구조

이 프로젝트의 `user` 서비스는 다음 두 파일로 구성되어 있습니다.

```text
UserService       (인터페이스)
UserServiceImpl   (구현체, @Service)
```

`UserService` (인터페이스):

```java
public interface UserService {
    String join(
        String id,
        String name,
        String email,
        String password
    );
}
```

- “회원 가입” 과 관련된 **기능 목록(계약)** 만 정의합니다.
- 컨트롤러는 이 인터페이스에만 의존하므로, 구현체가 바뀌어도 컨트롤러 코드는 바꿀 필요가 없습니다.

`UserServiceImpl` (구현체):

- `@Service`, `@RequiredArgsConstructor` 를 사용해 **실제 로직을 담는 클래스**입니다.
- 내부에서 `MemberRepository` 를 주입 받아, 회원 정보를 DB에 저장합니다.
- 비밀번호는 `join` 메서드 안에서 **SHA-256 해시**로 암호화한 후 저장하도록 구현되어 있습니다.

이런 식으로 **인터페이스(계약)와 구현체(로직)를 분리**해 두면:

- 나중에 구현을 교체하거나(예: 다른 암호화 방식, 외부 인증 서버 연동),
- 테스트에서 가짜 구현(Mock)을 주입해서 검증할 때도 훨씬 유연해집니다.
