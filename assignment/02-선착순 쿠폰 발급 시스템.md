# 2. 선착순 쿠폰 발급 시스템
## 문제 설명

한정 수량 쿠폰을 사용자에게 선착순으로 발급한다. 이미 발급받은 사용자는 같은 쿠폰을 다시 받을 수 없다.

## 요구사항

쿠폰 생성 API를 구현한다.

쿠폰 발급 API를 구현한다.

사용자가 발급받은 쿠폰 목록 조회 API를 구현한다.

한 쿠폰은 정해진 수량만 발급 가능해야 한다.

동일 사용자는 동일 쿠폰을 중복 발급받을 수 없다.

만료된 쿠폰은 발급할 수 없다.

쿠폰 상태는 발급가능, 소진, 만료 중 하나여야 한다.

## 확장 요구사항

선착순 동시 요청에 대해 정합성을 보장하라.

쿠폰 사용 API를 구현하고, 사용 후 재사용을 금지하라.

## 테스트 포인트

정상 발급

중복 발급 방지

수량 초과 방지

만료 쿠폰 발급 실패

## API 요청/응답 예시
요청/응답은 JSON을 기준으로 합니다. (`Content-Type: application/json`)

### 쿠폰 생성
- `POST /coupons`
요청(예):
```json
{
  "name": "WELCOME-5000",
  "totalQuantity": 100,
  "expiresAt": "2026-03-31T23:59:59Z"
}
```

응답(예):

```json
{
  "couponId": "c-1001",
  "name": "WELCOME-5000",
  "totalQuantity": 100,
  "issuedQuantity": 0,
  "expiresAt": "2026-03-31T23:59:59Z",
  "status": "AVAILABLE",
  "createdAt": "2026-03-23T09:00:00Z"
}
```

### 쿠폰 발급

* `POST /coupons/{couponId}/issue`
  요청(예):

```json
{
  "userId": "u-1001"
}
```

응답(예):

```json
{
  "userId": "u-1001",
  "couponId": "c-1001",
  "userCouponId": "uc-9001",
  "status": "ISSUED",
  "issuedAt": "2026-03-23T09:01:00Z",
  "expiresAt": "2026-03-31T23:59:59Z"
}
```

### 사용자 쿠폰 목록 조회

* `GET /users/{userId}/coupons`
  응답(예):

```json
{
  "userId": "u-1001",
  "coupons": [
    {
      "userCouponId": "uc-9001",
      "couponId": "c-1001",
      "name": "WELCOME-5000",
      "status": "ISSUED",
      "issuedAt": "2026-03-23T09:01:00Z",
      "usedAt": null,
      "expiresAt": "2026-03-31T23:59:59Z"
    }
  ]
}
```

### 쿠폰 사용

* `POST /users/{userId}/coupons/{userCouponId}/use`
  요청(예):

```json
{
  "orderId": "o-3001"
}
```

응답(예):

```json
{
  "userId": "u-1001",
  "userCouponId": "uc-9001",
  "couponId": "c-1001",
  "status": "USED",
  "usedAt": "2026-03-23T10:00:00Z"
}
```

### 공통 에러 응답(권장)

```json
{
  "errorCode": "COUPON_ALREADY_ISSUED",
  "message": "user already issued this coupon"
}
```

## 상세 요구사항(채점 기준용)

아래 항목을 충족하면 “문제 해결”로 인정합니다. (API 경로/응답 필드명은 예시이며, 핵심 규칙을 만족하면 됩니다.)

### 1) 쿠폰 생성

* 쿠폰 이름, 총 발급 수량(`totalQuantity`), 만료일시(`expiresAt`)를 받아 쿠폰을 생성합니다.
* 생성 시 상태는 현재 시점 기준으로 아래 규칙을 따릅니다.

    * 만료일시가 현재보다 과거면 `EXPIRED`
    * 발급 가능 수량이 남아 있고 만료되지 않았으면 `AVAILABLE`
* `totalQuantity`는 1 이상이어야 합니다.
* 예외:

    * `totalQuantity <= 0`이면 `400 Bad Request`
    * 필수값 누락 시 `400 Bad Request`

### 2) 쿠폰 발급

* 사용자는 특정 쿠폰을 1회만 발급받을 수 있습니다.
* 처리 성공 시:

    * 해당 사용자에게 `UserCoupon`이 1건 생성되어야 합니다.
    * 쿠폰의 발급 수량(`issuedQuantity`)이 1 증가해야 합니다.
    * 응답에는 최소한 `couponId`, `userId`, `userCouponId`, `issuedAt`, `expiresAt`이 포함되어야 합니다.
* 발급 가능 조건:

    * 쿠폰 상태가 `AVAILABLE`이어야 합니다.
    * 현재 시간이 `expiresAt` 이전이어야 합니다.
    * `issuedQuantity < totalQuantity`여야 합니다.
    * 동일 사용자가 동일 쿠폰을 이미 발급받지 않았어야 합니다.
* 예외:

    * 이미 발급받은 경우 `409 Conflict`
    * 수량이 모두 소진된 경우 `409 Conflict`
    * 만료된 쿠폰인 경우 `409 Conflict` 또는 `400 Bad Request`
    * 존재하지 않는 쿠폰이면 `404 Not Found`

### 3) 사용자 쿠폰 목록 조회

* `userId` 기준으로 사용자가 발급받은 쿠폰 목록을 조회합니다.
* 각 쿠폰에는 최소한 아래 정보가 포함되어야 합니다.

    * `userCouponId`
    * `couponId`
    * `name`
    * `status`
    * `issuedAt`
    * `usedAt`
    * `expiresAt`
* 정렬:

    * `issuedAt` 기준 내림차순(최신순) 조회를 권장합니다.

### 4) 쿠폰 상태 관리

* 쿠폰 상태는 아래 3가지 중 하나여야 합니다.

    * `AVAILABLE` : 발급 가능
    * `SOLD_OUT` : 수량 소진
    * `EXPIRED` : 만료
* 상태 판정 규칙 예시:

    * 현재 시간이 `expiresAt` 이상이면 `EXPIRED`
    * 현재 시간이 유효하더라도 `issuedQuantity >= totalQuantity`이면 `SOLD_OUT`
    * 그 외에는 `AVAILABLE`
* 상태는 조회 시점에 계산하거나, 저장된 값을 갱신하는 방식 중 하나를 일관되게 사용합니다.

### 5) 쿠폰 사용

* 사용자가 발급받은 쿠폰(`UserCoupon`)만 사용할 수 있어야 합니다.
* 처리 성공 시:

    * 사용자 쿠폰 상태가 `USED`로 변경되어야 합니다.
    * `usedAt`이 기록되어야 합니다.
* 사용 가능 조건:

    * 이미 사용된 쿠폰이 아니어야 합니다.
    * 만료되지 않은 쿠폰이어야 합니다.
* 예외:

    * 이미 사용된 쿠폰이면 `409 Conflict`
    * 만료된 쿠폰이면 `409 Conflict`
    * 사용자에게 없는 쿠폰이면 `404 Not Found`

## 확장 상세(가산점)

### A) 선착순 동시성 정합성

* 동시에 여러 사용자가 같은 쿠폰 발급을 요청하더라도 다음이 보장되어야 합니다.

    * 총 발급 수량을 초과해서 발급되지 않습니다.
    * 동일 사용자가 동일 쿠폰을 중복 발급받지 않습니다.
    * 최종 `issuedQuantity`와 실제 발급된 `UserCoupon` 수가 일치해야 합니다.
* 구현 방식 예시(택1 이상 권장):

    * 트랜잭션 + DB 락(비관적 락: `SELECT FOR UPDATE` 등)
    * 낙관적 락(버전 컬럼) + 재시도
    * 쿠폰 발급 이력에 `(userId, couponId)` 유니크 제약 추가
    * 수량 차감/발급 처리의 원자성 보장

### B) 쿠폰 사용 후 재사용 방지

* 동일 `userCouponId`에 대해 사용 요청이 여러 번 들어와도 한 번만 성공해야 합니다.
* 최초 사용 이후에는 상태가 `USED`로 유지되어야 하며, `usedAt`이 변경되면 안 됩니다.
* 필요 시 주문 번호(`orderId`) 기준 중복 사용 방지 정책을 추가할 수 있습니다.

## 테스트 시나리오(권장)

1. 정상 발급: 발급 가능한 쿠폰에 대해 최초 발급 성공, `issuedQuantity` 증가, 사용자 쿠폰 생성 확인
2. 중복 발급 방지: 같은 사용자가 같은 쿠폰을 2번 요청하면 1번만 성공하고 이후는 실패 확인
3. 수량 초과 방지: `totalQuantity`를 초과하는 수의 요청이 와도 성공 건수는 수량 이내인지 확인
4. 만료 쿠폰 발급 실패: `expiresAt`이 지난 쿠폰은 발급되지 않음을 확인
5. 쿠폰 사용 성공: 발급된 쿠폰 사용 시 상태가 `USED`로 변경되고 `usedAt` 저장 확인
6. 재사용 방지: 이미 사용한 쿠폰을 다시 사용하면 실패하고 상태가 유지되는지 확인
7. 동시성(확장): 멀티스레드 환경에서 동일 쿠폰 발급 요청 시 초과 발급/중복 발급이 발생하지 않는지 확인
