# 1. 포인트 지갑 시스템
## 문제 설명

사용자는 포인트를 충전하고, 사용할 수 있다. 모든 거래 내역은 저장되어야 하며, 잔액보다 많은 금액은 사용할 수 없다.

## 요구사항

사용자 포인트 잔액 조회 API를 구현한다.

포인트 충전 API를 구현한다.

포인트 사용 API를 구현한다.

포인트 거래 내역 조회 API를 구현한다.

잔액보다 큰 금액 사용 시 예외를 반환한다.

0 이하 금액 충전/사용은 허용하지 않는다.

거래 내역은 최신순으로 조회되어야 한다.

## 확장 요구사항

같은 요청이 중복 전송되면 한 번만 처리하도록 requestId 기반 중복 방지 기능을 추가하라.

동시 충전/동시 사용 시 정합성을 보장하라.

## API 요청/응답 예시
요청/응답은 JSON을 기준으로 합니다. (`Content-Type: application/json`)

### 잔액 조회
- `GET /point?userId={userId}`
응답(예):
```json
{
  "userId": "u-1001",
  "balance": 1200
}
```

### 충전
- `POST /point/charge`
요청(예):
```json
{
  "userId": "u-1001",
  "amount": 500,
  "requestId": "uuid-1234"
}
```
응답(예):
```json
{
  "userId": "u-1001",
  "balance": 1700,
  "transactionId": "t-999"
}
```

### 사용
- `POST /point/use`
요청(예):
```json
{
  "userId": "u-1001",
  "amount": 300,
  "requestId": "uuid-5678"
}
```
응답(예):
```json
{
  "userId": "u-1001",
  "balance": 1400,
  "transactionId": "t-888"
}
```

### 거래 내역 조회
- `GET point/transactions?limit=20&userId={userId}`
응답(예):
```json
{
  "userId": "u-1001",
  "transactions": [
    {
      "transactionId": "t-888",
      "type": "USE",
      "amount": 300,
      "balanceAfter": 1400,
      "createdAt": "2026-03-18T14:10:00Z"
    }
  ]
}
```

### 공통 에러 응답(권장)
```json
{
  "errorCode": "INVALID_AMOUNT",
  "message": "amount must be greater than 0"
}
```

## 상세 요구사항(채점 기준용)
아래 항목을 충족하면 “문제 해결”으로 인정합니다. (API 경로/응답 필드명은 예시이며, 핵심 규칙을 만족하면 됩니다.)

### 1) 잔액 조회
- `userId`에 대한 현재 `balance`를 조회합니다.
- 잔액이 존재하지 않는 사용자인 경우, 설계에 따라 `0`으로 초기화하거나 `404` 처리 중 한 가지로 일관되게 동작해야 합니다.

### 2) 포인트 충전
- 요청에 포함된 `amount`가 양수일 때만 처리합니다.
- 처리 성공 시:
  - 잔액이 `balance + amount`로 증가해야 합니다.
  - 거래 내역이 `CHARGE` 타입으로 1건 저장되어야 합니다.
  - 거래 내역에는 최소한 `type`, `amount`, `balanceAfter`, `createdAt`이 포함되어야 합니다.
- 예외:
  - `amount <= 0`이면 `400 Bad Request`

### 3) 포인트 사용
- 요청에 포함된 `amount`가 양수일 때만 처리합니다.
- 처리 성공 시:
  - 잔액이 `balance - amount`로 감소해야 합니다.
  - 거래 내역이 `USE` 타입으로 1건 저장되어야 합니다.
- 예외:
  - `amount <= 0`이면 `400 Bad Request`
  - `balance < amount`이면 `409 Conflict` (또는 비슷한 비즈니스 예외)

### 4) 거래 내역 조회
- `userId` 기준으로 거래 내역을 조회합니다.
- 정렬:
  - `createdAt` 기준 내림차순(최신순)으로 반환해야 합니다.
- 페이지네이션:
  - `limit` 같은 간단한 파라미터를 제공하면 가산점입니다. (필수는 설계에 따라 조정)

## 확장 상세(가산점)

### A) requestId 기반 중복 방지(멱등성)
- 충전/사용 요청 모두 `requestId`를 포함한다고 가정합니다.
- 동일한 `userId + requestId`가 재전송되어도:
  - 잔액은 한 번만 반영되어야 합니다.
  - `PointTransaction`은 중복 저장되면 안 됩니다.
- 구현 방식 예시(택1):
  - `ProcessedRequest` 테이블에 `userId + requestId`를 유니크 키로 저장하고, 이미 처리된 경우 이전 결과를 반환

### B) 동시성 정합성
- 동시에 여러 요청이 들어와도 다음이 보장되어야 합니다.
  - 잔액은 음수가 되지 않습니다.
  - 최종 잔액과 거래 내역(`balanceAfter`)이 논리적으로 일관됩니다.
- 구현 방식 예시(택1 이상 권장):
  - 트랜잭션 + DB 락(비관적 락: `SELECT FOR UPDATE` 등)
  - 낙관적 락(버전 컬럼) + 재시도

## 테스트 시나리오(권장)
1. 충전 성공: 양수 amount로 잔액 증가와 거래내역 1건 저장 확인
2. 사용 성공: 잔액 충분 상태에서 잔액 감소와 거래내역 1건 저장 확인
3. 잔액 부족 실패: 예외 발생 + 잔액/거래내역 변경 없음 확인
4. 중복 요청: 동일 `userId + requestId`로 2번 호출 시 잔액/거래내역이 1번만 반영됨 확인
5. 동시성(확장): 멀티스레드로 USE를 동시에 호출해도 잔액 음수 불가 + 최종 잔액 일관성 확인
